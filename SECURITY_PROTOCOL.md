## Part 1 — 위협 모델

### 1.1 공격자 유형

| 유형 | 동기 | 주요 공격 벡터 |
|------|------|----------------|
| 금융 사기꾼 | GDC 탈취·세탁 | 결제 위조, 계정 도용 |
| 신원 도용자 | 타인 계정 이용 | JWT 탈취, GUID 충돌 |
| 사기 업체 | 허위 거래 | 유령 업체, 허위 리뷰 |
| 스패머 | 광고·피싱 | 게시판·AI 악용 |
| 내부자 | 정보 탈취 | DB 직접 접근 |
| 국가 행위자 | 감시·교란 | 인프라 공격 |
| 자동화 봇 | 무차별 공격 | Rate Limit 우회 |

### 1.2 고팡 특유의 위협

```
다국어 환경:
  번역 파이프라인을 통한 인젝션
  언어 감지 우회로 필터 회피

외국인 관광객 표적:
  낯선 환경의 취약성 악용
  긴급 상황 위장 사기

GDC 특수성:
  법정화폐 아님 → 분쟁 해결 복잡
  익명성 악용 자금 세탁

AI 비서 악용:
  프롬프트 인젝션
  AI를 통한 개인정보 탈취
  AI↔AI 소통의 악의적 자동화
```

### 1.3 보호 대상 자산

```
최우선:
  fs_ledger (금융 원장) — 위변조 시 치명적
  private 계층 JSONB    — 개인정보 유출
  JWT 서명 키           — 탈취 시 전체 인증 붕괴
  AES 암호화 키         — 탈취 시 LLM 키 유출

중요:
  GUID 결정성           — 충돌 시 거래 혼선
  PDV Hash Chain        — 위변조 시 감사 불가
  AI 시스템 프롬프트    — 유출 시 업체 정보 노출

일반:
  public 계층 JSONB     — 유출되어도 피해 제한적
  커뮤니티 게시글       — 공개 정보
```

## Part 2 — 사전 예방 계층

### 2.1 인증 강화

```
JWT 설계:
  알고리즘: HMAC-SHA256
  만료: 24시간
  갱신 허용: 만료 1시간 전 구간만
  페이로드: guid, name, lang, type, iat, exp
  저장: localStorage (httpOnly 쿠키 검토 v2.0)

timingSafeEqual:
  서명 검증 시 반드시 사용
  문자열 비교 금지 (타이밍 어택 방지)

JWT 무효화:
  현재: 만료 전 무효화 불가 (stateless)
  v1.1: Redis/KV 기반 블랙리스트 도입
```

### 2.2 Rate Limiting

```
엔드포인트별 제한 (Cloudflare Worker):

/register-consumer:  IP당 10회/시간
/register:           IP당 5회/시간
/ai-chat:            GUID당 100회/시간, IP당 200회/시간
/biz/order:          GUID당 20회/시간
/community POST:     GUID당 10회/시간
/review POST:        GUID당 5회/시간 (tx_id 1건당 1회)
/nearby:             IP당 60회/시간
/search:             IP당 120회/시간
/interpret:          IP당 300회/시간

초과 시: 429 Too Many Requests + Retry-After 헤더
누적 위반: IP 임시 차단 (1시간)
```

### 2.3 사칭 방지 handle 정책

```
보호 handle 목록 (등록 불가):
  공공기관: @*_cityhall, @*_police, @*_fire,
            @*_hospital (기관만 허용), @*_school
  고팡 시스템: @gopang_*, @admin_*, @system_*
  유명 브랜드: @*_samsung, @*_kakao 등 (목록 관리)

검증 절차:
  institution 등록 시 → trust_level 3 필요
  관리자 수동 승인 후 보호 handle 부여
```

### 2.4 입력값 검증

```
모든 API 입력:
  SQL 인젝션: Supabase 파라미터 바인딩 사용 (문자열 연결 금지)
  XSS:        텍스트 필드 HTML 이스케이프
  JSONB:      크기 제한 (extra 최대 64KB)
  파일 업로드: 허용 MIME 타입만 (이미지: jpg/png/webp)
  URL:        외부 URL은 화이트리스트 도메인만 허용

전화번호:
  숫자만 추출 후 GUID 생성
  원본 평문 저장 금지

금액:
  음수 거부
  최대 단일 거래액: ₮10,000,000
  일일 거래 한도: ₮50,000,000 (v1.0 기본값)
```

## Part 3 — 실시간 탐지 계층

### 3.1 K-Security 스코어링 (기존 확장)

```
현재 구현:
  1단계: 로컬 패턴 (스팸·혐오)
  2단계: Anthropic 정밀 분석 (0.6~0.85 구간)

추가 탐지 패턴:
  피싱:      http(s):// 포함 + 클릭 유도 문구
  개인정보:  주민번호 패턴, 계좌번호 패턴
  인젝션:    시스템 프롬프트 조작 시도
  사기 문구: "무료", "당첨", "송금해주세요"
  긴급 위장: 공공기관 사칭 문구

임계값:
  score < 0.6:  정상 → 즉시 처리
  0.6 ≤ score < 0.85: S2 → 지연 처리 + 검토 플래그
  score ≥ 0.85: S3 → 즉시 차단 + 관리자 알림
```

### 3.2 이상 거래 탐지 (FDS)

```
탐지 규칙:

R1 — 속도 이상:
  1분 내 동일 GUID 3건 이상 결제 → 일시 중단

R2 — 금액 이상:
  평소 거래액의 10배 초과 → 추가 확인 요청

R3 — 패턴 이상:
  새벽 2~5시 집중 거래 → 플래그

R4 — 위치 이상:
  GPS 위치와 거래 업체 위치 50km 이상 차이 → 경고

R5 — 신규 계정 이상:
  등록 후 1시간 내 ₮100,000 이상 거래 → 지연 처리

R6 — 순환 거래:
  A→B→C→A 형태 단기 순환 → 자금 세탁 의심

탐지 시:
  즉시 차단 없음 (오탐 방지)
  → 소비자에게 "이례적 거래" 확인 알림
  → 확인 후 진행 또는 취소
  → 3회 미확인 → 관리자 검토
```

### 3.3 이상 로그인 탐지

```
탐지 조건:
  - 24시간 내 5개 이상 다른 IP에서 로그인
  - 불가능한 이동 (서울 로그인 1분 후 제주 로그인)
  - 알려진 악성 IP/VPN 대역

탐지 시:
  → 추가 인증 요청 (전화번호 재확인)
  → 본인 미확인 시 JWT 발급 거부
  → security_event INSERT
```

### 3.4 실시간 모니터링 대시보드 (v1.1)

```
관리자 대시보드 표시 항목:
  - 시간당 신고 건수
  - FDS 플래그 건수
  - K-Security S2/S3 발생 현황
  - Rate Limit 초과 IP 목록
  - 에스컬레이션 미처리 건수
```

## Part 4 — 사후 대응 계층

### 4.1 사고 대응 절차 (IRP)

```
Level 1 — 경미 (개별 사용자 피해)
  탐지 → 자동 조치 → 24시간 내 사후 보고

Level 2 — 중대 (다수 사용자 또는 금융 피해)
  탐지 → 즉시 관리자 알림 → 1시간 내 조치
  → 피해자 개별 통보 → 72시간 내 원인 분석

Level 3 — 심각 (시스템 전체 위협)
  탐지 → 즉시 서비스 일부 중단 검토
  → 전체 사용자 공지 → 수사기관 협조
```

### 4.2 계정 조치 단계

```
경고(1회):  위반 행위 중단 요청 + 기록
정지(임시): 24시간~7일 서비스 이용 중단
            → is_active=false + lifecycle.inactive_reason
정지(장기): 30일~6개월
영구 정지:  재등록 방지 → id_number_hash 블랙리스트
            is_public=false + deleted_at 기록

이의신청:
  조치 통보 후 7일 내 이의신청 가능
  관리자 재검토 → 48시간 내 결정
```

### 4.3 피해자 구제

```
결제 사기 피해:
  1. 피해 신고 접수
  2. fs_ledger 거래 검토 (불변 원장 기준)
  3. 사기 확인 시 → 플랫폼 보증 환불 (GDC)
  4. 사기범 계정 영구 정지 + 잔액 동결
  5. 피해 규모 > ₮1,000,000 → 수사기관 의뢰

개인정보 유출:
  1. 72시간 내 피해자 개별 통보
  2. 관련 세션 무효화
  3. 비밀번호·키 갱신 안내
  4. 개인정보보호위원회 신고 (법적 의무)
```

### 4.4 블랙리스트 관리

```
블랙리스트 항목:
  guid_blacklist:   영구 정지 GUID
  ip_blacklist:     악성 IP 대역
  phone_hash_blacklist: 사기 전화번호 해시
  handle_reserved:  보호 handle 목록

저장 위치:
  Cloudflare KV (빠른 조회)
  Supabase security_log (감사 기록)

갱신 주기:
  자동: FDS·K-Security S3 발생 시 즉시
  수동: 관리자 주 1회 검토
```

## Part 5 — 금융 보안

### 5.1 결제 흐름 보안

```
결제 요청 검증 순서:
  1. JWT 유효성 검증
  2. QR created_at + expires 만료 검증
  3. 구매자 GDC 잔액 충분 여부 확인
  4. tx_id 중복 여부 확인 (UNIQUE 제약)
  5. FDS 규칙 검사
  6. ED25519 서명 검증 (v1.1)
  7. fs_ledger 3행 원자 INSERT
  8. L1 블록 기록

실패 시 전체 롤백 (트랜잭션)
```

### 5.2 GDC 잔액 보호

```
잔액 조작 방지:
  extra.private.finance.fs 직접 UPDATE 금지
  반드시 fs_ledger INSERT 후 RPC로 집계
  관리자도 직접 수정 불가 → 로그 남는 RPC만

한도 관리:
  단일 결제 최대: ₮10,000,000
  일일 송금 최대: ₮50,000,000
  한도 초과 시: 관리자 승인 필요

잔액 음수 방지:
  결제 전 잔액 확인 필수
  음수가 되는 결제 거부
  (단, 신용 한도 설정 시 예외 — v2.0)
```

### 5.3 자금 세탁 방지 (AML)

```
의심 거래 패턴:
  - 단기 순환 거래 (A→B→C→A)
  - 다수 소액 분산 → 통합 (스머핑)
  - 신규 계정 간 대량 거래
  - 거래 목적 불분명 (업종과 무관한 거래)

보고 기준:
  단일 거래 ₮5,000,000 이상: 기록 강화
  동일인 일 누계 ₮10,000,000 이상: 관리자 검토
  의심 확정 시: 금융정보분석원(KoFIU) 보고 검토

GDC의 법적 지위:
  현재: 포인트/상품권에 준하는 처리
  v2.0: 법적 자문 후 전자지급수단 여부 결정
```

## Part 6 — 신원 보안

### 6.1 GUID 충돌 방지

```
uuidv5 충돌 가능성:
  이론적: SHA-1 기반, 충돌 확률 극히 낮음
  실제 위험: 전화번호 재활용
    → 같은 번호를 다른 사람이 사용할 경우
    → auth.verified_at 타임스탬프로 최신 소유자 구분

대응:
  동일 GUID 신규 등록 → 기존 계정 점유 확인
  점유자 없음: 신규 등록 허용
  점유자 있음: 이전 소유자에게 알림 + 확인 절차
```

### 6.2 계정 복구 vs 도용 구분

```
정당한 복구:
  동일 전화번호 → 동일 GUID 자동 복구
  거래 이력 자동 연결

도용 의심:
  복구 시 기존 기기와 완전히 다른 환경
  → 기존 등록 전화번호로 확인 요청
  → 확인 불가 시 → 관리자 수동 심사

복구 후:
  이전 모든 JWT 무효화
  로그인 이력 알림 (보안 이메일)
```

### 6.3 handle 보안

```
handle 변경:
  30일에 1회만 허용
  이전 handle → 새 handle 리다이렉트 유지 (6개월)
  이전 handle 재사용 금지 (영구)

handle 탈취 방지:
  유명 업체 handle 선점 → 관리자 신고 후 회수
  보호 handle 등록 불가 (§2.3)
```

## Part 7 — 데이터 보안

### 7.1 계층별 암호화

```
저장 암호화 (at rest):
  Supabase: AES-256 (플랫폼 기본)
  민감 JSONB (S09 Health, 계좌):
    추가 AES-256-GCM 레이어
    키: AES_ENCRYPTION_KEY (Cloudflare 환경변수)

전송 암호화 (in transit):
  모든 API: TLS 1.3
  Worker ↔ Supabase: HTTPS
  Supabase Realtime: WSS

키 관리:
  GOPANG_MASTER_KEY:    JWT 서명 (Cloudflare 환경변수)
  AES_ENCRYPTION_KEY:   JSONB 암호화 (Cloudflare 환경변수)
  SUPABASE_SERVICE_KEY: DB 접근 (Cloudflare 환경변수)
  키 교체 주기: 1년 (침해 시 즉시)
```

### 7.2 개인정보 최소화 원칙

```
수집 최소화:
  등록 필수: 전화번호(GUID 생성용) + 이름
  수집 후 전화번호 평문 저장 금지
  → 숫자만 추출 → GUID 생성 → 해시만 저장

보존 최소화:
  음성 원본: STT 변환 후 즉시 폐기
  로그인 이력: 90일 자동 삭제
  AI 대화: 30일 자동 삭제 (결제 관련 제외)
  GPS 정밀 좌표: 소비자는 100m 반올림

접근 최소화:
  Worker: 필요한 컬럼만 SELECT
  관리자: 업무 목적 외 조회 금지 + 감사 로그
```
